Research

Academic and institutional research on agentic commerce and payments, from arXiv, SSRN and major institutions.

arXiv - 11 papers
How Agentic Is Agentic Commerce? (x402 adoption measurement)
arXiv 2607.12575 · Jul 14, 2026
Population-scale measurement of x402 on Base: 136.7M settlements, but 85% fabricated or internal to a single cluster. The headline transaction counter measures manufacturability, not real adoption.
The Web4 Agent Economy
arXiv 2606.25876 · Jun 24, 2026
Large-scale empirical study of the protocol landscape (MCP, x402, ERC-8004) that lets agents hold a crypto wallet, execute on-chain transactions, and pay for external API calls.
Can Trustless Agents Be Trusted? An Empirical Study of ERC-8004
arXiv 2606.26028 · Jun 2026
Empirical audit of the decentralized ERC-8004 "Trustless Agents" ecosystem - tests whether the registry actually delivers the trust guarantees it advertises.
Agent-to-Agent Finance: Blockchain Payments and Trust Infrastructure for Autonomous AI Agents
arXiv 2607.00245 · Jul 1, 2026
Surveys how AP2, x402 and ERC-8004 combine to let autonomous agents negotiate, pay and manage digital assets without human sign-off at each step.
Free-Riding the Agentic Web: A Systematic Security Analysis of x402 Payments
arXiv 2605.30998 · May 2026
Catalogs exploit paths that let an agent consume a paid API without actually settling payment - directly relevant to anyone treating x402 receipts as proof of service delivery.
SoK: Security of Autonomous LLM Agents in Agentic Commerce
arXiv 2604.15367 · Apr 2026
Systematization-of-knowledge threat model spanning MCP, A2A, Agora and ANP - the closest thing to a shared vocabulary for "what can go wrong" across the whole protocol stack.
SoK: Blockchain Agent-to-Agent Payments
arXiv 2604.03733 · Apr 2026
Companion systematization focused specifically on the blockchain-settled leg of agent-to-agent commerce.
TessPay: Verify-then-Pay Infrastructure for Trusted Agentic Commerce
arXiv 2602.00213 · Feb 2026
Escrow-based design: funds lock until cryptographic proof of task execution exists, and release only after independent verification - a concrete implementation of the "verify before you pay" pattern FACT and ATLAS also target.
RAILS: Verification-Native Clearing for Agentic Commerce
arXiv 2606.08790 · Jun 2026
Argues MCP, A2A, x402, AP2 and the Visa/Mastercard networks all assume a neutral mechanism decides whether an agent fulfilled its mandate - but none actually provides one. Directly echoes the problem trust layers like FACT and ATLAS are built to solve.
Paying to Know: Micro-Transaction Markets for Verified Product Information in Agentic E-Commerce
arXiv 2606.24783 · Jun 2026
Proposes a micropayment market for verified product data itself - a shopping agent pays a small fee to confirm a listing is accurate before it transacts on it.
Distilling a Shopping Agent from Arena Traces
arXiv 2606.10064 · Jun 8, 2026
Uses traces from a Bittensor "arena" (ShoppingBench) to train small agentic shopping models, addressing the shortage of reliable, unbiased training trajectories for this task.

Major protocols

the ones to focus on, with a direct link to their official spec
Every link below goes to the protocol owner's own documentation, GitHub repository or developer portal, not a third-party summary.
AP2 · Agent Payments Protocol · Google, donated to FIDO Alliance Specification →

Cryptographically-signed "Mandates" for delegated, verifiable agent payments. Open extension of A2A and MCP.

x402 · Coinbase, governed by the x402 Foundation (Linux Foundation) Specification →

HTTP-native micropayments reviving the dormant "402 Payment Required" status code, settled in stablecoins.

ACP · Agentic Commerce Protocol · OpenAI & Stripe Specification →

Open standard connecting buyers, agents and businesses to complete purchases; powers Instant Checkout in ChatGPT.

UCP · Universal Commerce Protocol · Google, Shopify + 20 partners Specification →

Vendor-neutral standard modeling the full shopping journey (checkout, identity linking, order lifecycle), not just payment.

MCP · Model Context Protocol · Anthropic, hosted by the Linux Foundation Specification →

Standardizes how agents reach external tools, APIs and data. The plumbing most commerce protocols build on top of.

A2A · Agent2Agent Protocol · Google, transferred to the Linux Foundation Specification →

How independent agents discover each other, negotiate, and exchange tasks and results as peers.

ERC-8004 · Trustless Agents · Ethereum Specification →

On-chain Identity, Reputation and Validation registries letting agents check counterparties before transacting.

Visa Trusted Agent Protocol · Visa Specification →

Cryptographically signed HTTP messages (RFC 9421) to recognize Visa-approved agents and validate their intent at checkout.

FACT - Framework for Agentic Commerce Trust

Fime launches FACT, a neutral, independent trust layer between AI systems and payment rails: intent validation, continuous compliance monitoring, independent auditor agents, and transaction-by-transaction trust attestation. Unlike mechanisms built into a single network or platform, FACT stays interoperable and free of conflicts of interest.

"We've built systems that let AI transact - we haven't yet built the systems that let us trust those transactions at scale." Lionel Grosclaude, CEO of Fime
Full announcement - fime.com Coverage - The Paypers Coverage - Biometric Update Coverage - FFNews

FACT's four functions

01

Intent validation

Checks that the action an agent proposes stays aligned with the intent expressed by the user or business.

02

Real-time compliance

Continuous monitoring of agent behavior against defined rules and regulatory requirements.

03

Independent auditor agents

Neutral third-party agents provide impartial trust verification, accredited by a certification body.

04

Transaction-by-transaction attestation

A per-transaction trust signal usable for decisioning, auditability and traceability.

What FACT adds on top of Verifiable Intent

Verifiable Intent (VI), Mastercard/Google's cryptographic protocol, proves a transaction satisfies the formal constraints of a signed mandate - a precise, binary result, but blind to everything that happened before it. FACT continuously observes the agent's exchanges (prompts, tool calls, cart context) and compares them against the user's original intent: a transaction can be formally valid under VI while still resulting from a misread intent or a prompt injection.

Verifiable Intent

Verifies the outcome: does the transaction respect the constraints the user signed (merchant, amount, items)?

FACT

Verifies the behavior: did the agent's reasoning stay faithful to the user's intent throughout the negotiation?

Why an independent layer

FACT is designed as a third-party layer, outside payment networks and agent platforms, so it can stay interoperable and neutral. That independence is what lets merchants, issuers and regulators accept the same trust signal regardless of which payment protocol runs upstream (AP2, ACP, UCP, x402...). It's also what distinguishes FACT from trust mechanisms built directly into a single network or platform, which can't offer the same neutrality.

Competitive watch

FACT isn't the only proposal trying to answer "what did the agent actually do." Since early June 2026, several research and standards efforts have converged on the same problem from different angles. Only entries below have a real, checkable paper or draft behind them, anything we couldn't verify (including a couple of names on an initial researcher's list, see the Perplexity page) was left off rather than guessed at.

RAILS · arXiv preprint · Jun 2026 arXiv 2606.08790 →

The closest direct conceptual competitor: seven primitives (Obligation Object, Evidence Envelope, Verification Mesh, Clearing Decision, Settlement Instruction...) built to produce an actual clearing decision, not just a compliance record. Where FACT attests trust, RAILS argues you need a decision a settlement system can act on.

Notarized Agents · arXiv preprint · Jun 2026 arXiv 2606.04193 →

Inverts the trust boundary: the service receiving an agent's call signs its own receipt, encrypts it to the agent's owner, and publishes it to a witness-cosigned transparency log. Challenges the need for a central auditor like FACT by having every counterparty self-attest instead.

EMILIA Protocol · open-source, active IETF Internet-Drafts emiliaprotocol.ai →

A named human signs the exact high-risk action before it runs (payment, deploy, delete); the receipt is offline-verifiable and formally modeled (TLA+/Alloy). Overlaps FACT on pre-action authorization proof, but stops at the authorization moment rather than monitoring the agent's reasoning throughout.

WIMSE Applicability for AI Agents · IETF Internet-Draft IETF Datatracker →

Extends the IETF's Workload Identity in Multi-System Environments work to AI agents, binding an agent's cryptographic identity to its owner. A potential standardization path for the "which agent instance did this" question FACT and ATLAS both need answered.

Complementary technologies

These don't compete with FACT, they solve adjacent problems FACT could plug into rather than rebuild.

AI Agent Authorization Integration Framework · IETF Internet-Draft, authors from Alibaba, Cisco, Okta IETF Datatracker →

Assembles existing OAuth 2.0, OpenID Connect and WIMSE building blocks into cross-domain identity, policy and multi-hop delegation for agents. Doesn't invent a new trust mechanism, but could hand FACT a ready-made identity chain to audit against.

Agent Authorization Profile (AAP) for OAuth 2.0 · IETF Internet-Draft IETF Datatracker →

Structured OAuth/JWT claims for agent identity, task context, delegation chains and human-oversight requirements. Gives FACT a standard place to read "who authorized this agent to do what" instead of parsing bespoke tokens per platform.

ERC-8004 · Ethereum standard eips.ethereum.org →

On-chain Identity, Reputation and Validation registries. Complements FACT's per-transaction attestation with a persistent, portable reputation record an auditor agent could check before it even starts monitoring.

Hyperlab Fime Working draft, v0.1 - July 2026, not yet endorsed by an external standards body

ATLAS - Agent Trust Layer and Assurance Standard

A cryptographic protocol for independent assessment of autonomous AI agent behavior in commerce transactions. ATLAS standardizes how "Assessor" agents obtain encrypted evidence from subject agents, evaluate their compliance with policy, and produce verifiable trust signals for payment systems - developed by Fime/Consult Hyperion's Hyperlab as part of a Continuous Certification programme for agentic commerce.

Full specification - hyperlab-fime.github.io

The problem: intent drift

An AI agent's interpretation of user constraints - budget, allergens, preferences - can diverge from actual intent across the interpretation, structuring and filtering layers it passes through before acting. No standardized mechanism currently exists for independent, runtime verification of what an agent actually did during a transaction, which leaves gaps in liability allocation, fraud detection and regulatory compliance.

Five linked artifacts

Link 1

Assessment delegation

The subject agent delegates assessment of its own behavior.

Link 2

Sub-delegation

The holder's sub-delegation, used in merchant / credential-provider mode only.

Link 3

Access assertion

The Assessor's access assertion, carrying its own authorization credentials.

Link 4

Verdict

The Assessor's verdict, with evidence hashes and a report token.

Link 5

Redemption token

An opaque downstream token for payment rails, carrying no raw intent data.

Two operational patterns: holder mode, where a merchant or credential provider commissions the assessment, and direct mode, where the agent self-commissions it for compliance purposes.

How it relates to AP2, Verifiable Intent and FACT

ATLAS complements rather than replaces these - it adds a layer none of them currently provide.

AP2 / Verifiable Intent

Secure mandate execution: does the transaction match the cryptographically signed constraints the user agreed to?

ATLAS

Secures runtime behavior: continuously monitors what the agent actually does versus what it claims to do, independent of the payment rail underneath.

Evidence stays encrypted (JWE) to Assessors only, using JWS/JWT capability tokens with RFC 7800 key binding (cnf claims) over TLS transport. Downstream payment systems only ever receive the compact opaque Link 5 token, never raw intent data - the same neutrality principle FACT applies to its own attestations.
